# Reproducibility Experiment v2

> **Goal:** Quantitatively measure whether NAIL's canonical form improves structural reproducibility compared to Python for non-trivial functions.  
> **Result:** Hypothesis rejected ŌĆö see [Actual Results](#actual-results-2026-02-24-n5-per-language-per-problem).

## Experiment Purpose

This experiment tests whether NAIL's JSON canonical form (JCS) leads to higher **output reproducibility** when the same function specification is given to an LLM multiple times.

### Hypothesis

Because NAIL:
1. Has a single, rigid JSON representation (no syntactic sugar, no style choices)
2. Enforces JCS canonical form (sorted keys, no whitespace)
3. Uses a fixed set of operators with fixed JSON schemas

ŌĆ”an LLM generating NAIL code for the same spec repeatedly should converge to a small number of unique structural forms.

Python, by contrast, offers freedom in variable naming, commenting style, blank lines, and algorithmic expression ŌĆö leading to higher output variance.

## Differences from v1

| Aspect | v1 | v2 |
|--------|----|----|
| Task | `add(a, b)` ŌĆö trivial | `clamp`, `fibonacci`, `fizzbuzz_count` ŌĆö requires control flow |
| NAIL prompt | **Includes concrete body example** (leaks the answer) | Schema structure only ŌĆö no body examples |
| Convergence reason | LLM copies the given answer | LLM must reason from spec to implementation |
| Problem count | 1 | 3 |
| Runs per problem | 10 | 5 per language per problem |
| Result format | Flat `results.json` | Per-problem grouped `results_v2.json` |

### v1 Root Problems

1. **Task too easy** ŌĆö `add(a, b)` returns `a + b` in any language. There is essentially one possible implementation. Even Python will converge.

2. **Answer leaked in NAIL prompt** ŌĆö The v1 NAIL prompt included:
   ```
   "- to return a+b: [{"op": "return", "val": {"op": "+", "l": {"ref": "a"}, "r": {"ref": "b"}}}]"
   ```
   This meant the LLM wasn't generating code ŌĆö it was copying a template.

### v2 Fixes

- **Harder tasks** require control flow: `if`, `loop`, `let`, `assign`
- **NAIL prompt contains schema description only** ŌĆö operators listed by type, no worked examples
- All three tasks require the LLM to derive the implementation from the spec alone

## Problems

### `clamp(val, lo, hi) -> int64`
Return `val` clamped to the range `[lo, hi]`.

Requires: nested `if` statements or `min`/`max` equivalents.

### `fibonacci(n) -> int64`
Return the n-th Fibonacci number (0-indexed). `fibonacci(0) = 0`, `fibonacci(1) = 1`.

Requires: `loop` with mutable variables, `let mut` + `assign`.

Note: NAIL `kind: "fn"` cannot call other functions, and recursion is forbidden ŌĆö so iteration is the only valid approach.

### `fizzbuzz_count(n) -> int64`
Return how many numbers from 1..n are divisible by 3 or 5.

Requires: `loop` with counter, modulo operator `%`, conditional `if`.

## Expected Results

| Problem | NAIL unique hashes | Python unique hashes |
|---------|--------------------|----------------------|
| `clamp` | 2ŌĆō4 | 5ŌĆō8 |
| `fibonacci` | 2ŌĆō4 | 5ŌĆō10 |
| `fizzbuzz_count` | 3ŌĆō5 | 6ŌĆō10 |

NAIL variance is expected to come from:
- Condition ordering in nested `if` (e.g., check `lo` first vs. `hi` first)
- Loop variable name choice (e.g., `i` vs. `idx`)
- Mutable variable naming in fibonacci (e.g., `a`/`b` vs. `prev`/`curr`)

Python variance comes from variable names, docstrings, comments, blank lines, and algorithmic style.

## Metric Definitions

**`match_rate`** (reported as `MatchRate` in `analyze.py`):  
The fraction of runs whose output matches the single most-common output hash.  
Formula: `count_of_most_frequent_hash / total_runs`.  
Example: if 10 runs produce hashes `[A, A, A, B, C]`, match_rate = 3/5 = 60%.  
Interpretation: 100% = fully deterministic; 20% (1-of-5) Ōēł random for the problem size.

**`unique`**: Number of distinct output hashes across all runs for that problem+language.  
Lower = more reproducible.

**`valid`** (NAIL only): Number of runs where the output passed the NAIL checker (L0ŌĆōL2).  
Python validity is not checked (no static guarantees).

## Methodology Notes

- NAIL outputs are **normalized to JCS canonical form** (`json.dumps(sort_keys=True, separators=(',', ':'))`) before hashing ŌĆö this removes key-order variance, which is not semantically meaningful.
- NAIL outputs are also **type-checked** using the NAIL interpreter's `Checker`. Invalid outputs are flagged but still counted in hash diversity.
- Python outputs are hashed as-is (raw text). No normalization is applied, which makes the comparison conservative (favoring Python).

## Actual Results (2026-02-24, n=5 per language per problem)

| Problem | NAIL unique/runs | NAIL valid | NAIL match_rate | Python unique/runs | Python match_rate |
|---------|-----------------|-----------|-----------------|-------------------|-------------------|
| `clamp` | 1/5 | 5/5 | **100%** | 1/5 | **100%** |
| `fibonacci` | 5/5 | 5/5 | 20% | 3/5 | 60% |
| `fizzbuzz_count` | 2/5 | 5/5 | 60% | 1/5 | **100%** |

**Key finding:** The hypothesis was **rejected**. Python shows higher overall reproducibility (5 unique/15 total) than NAIL (8 unique/15 total).

**Interpretation:**  
- `clamp` (single algorithmic path): both converge to 100% ŌĆö expected.  
- `fibonacci` / `fizzbuzz_count` (multiple valid implementations): Python converges because the canonical implementation exists in abundant training data. NAIL has no such prior ŌĆö but all 15/15 NAIL outputs passed the checker.  
- JCS eliminates **syntactic** variance (key ordering, whitespace). It does not eliminate **algorithmic** variance (loop structure, variable naming). This is the cold-start limitation of a new language.  
- **valid 15/15 is the real result**: NAIL's verifiability guarantee holds regardless of convergence.
