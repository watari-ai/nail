# Reproducibility Experiment v2

> **Goal:** Quantitatively demonstrate that NAIL's canonical form improves structural reproducibility compared to Python for non-trivial functions.

---

## Experiment Purpose

This experiment tests whether NAIL's JSON canonical form (JCS) leads to higher **output reproducibility** when the same function specification is given to an LLM multiple times.

> *NAILのcanonical formが構造的再現性を高めることを定量的に示す*

### Hypothesis

Because NAIL:
1. Has a single, rigid JSON representation (no syntactic sugar, no style choices)
2. Enforces JCS canonical form (sorted keys, no whitespace)
3. Uses a fixed set of operators with fixed JSON schemas

…an LLM generating NAIL code for the same spec repeatedly should converge to a small number of unique structural forms.

Python, by contrast, offers freedom in variable naming, commenting style, blank lines, and algorithmic expression — leading to higher output variance.

---

## Differences from v1

| Aspect | v1 | v2 |
|--------|----|----|
| Task | `add(a, b)` — trivial | `clamp`, `fibonacci`, `fizzbuzz_count` — requires control flow |
| NAIL prompt | **Includes concrete body example** (leaks the answer) | Schema structure only — no body examples |
| Convergence reason | LLM copies the given answer | LLM must reason from spec to implementation |
| Problem count | 1 | 3 |
| Runs per problem | 10 | 10 per language per problem |
| Result format | Flat `results.json` | Per-problem grouped `results_v2.json` |

### v1 Root Problems

1. **Task too easy** — `add(a, b)` returns `a + b` in any language. There is essentially one possible implementation. Even Python will converge.

2. **Answer leaked in NAIL prompt** — The v1 NAIL prompt included:
   ```
   "- to return a+b: [{"op": "return", "val": {"op": "+", "l": {"ref": "a"}, "r": {"ref": "b"}}}]"
   ```
   This meant the LLM wasn't generating code — it was copying a template.

### v2 Fixes

- **Harder tasks** require control flow: `if`, `loop`, `let`, `assign`
- **NAIL prompt contains schema description only** — operators listed by type, no worked examples
- All three tasks require the LLM to derive the implementation from the spec alone

---

## Problems

### `clamp(val, lo, hi) -> int64`
Return `val` clamped to the range `[lo, hi]`.

Requires: nested `if` statements or `min`/`max` equivalents.

### `fibonacci(n) -> int64`
Return the n-th Fibonacci number (0-indexed). `fibonacci(0) = 0`, `fibonacci(1) = 1`.

Requires: `loop` with mutable variables, `let mut` + `assign`.

Note: NAIL `kind: "fn"` cannot call other functions, and recursion is forbidden — so iteration is the only valid approach.

### `fizzbuzz_count(n) -> int64`
Return how many numbers from 1..n are divisible by 3 or 5.

Requires: `loop` with counter, modulo operator `%`, conditional `if`.

## Expected Results

| Problem | NAIL unique hashes | Python unique hashes |
|---------|--------------------|----------------------|
| `clamp` | 2–4 | 5–8 |
| `fibonacci` | 2–4 | 5–10 |
| `fizzbuzz_count` | 3–5 | 6–10 |

NAIL variance is expected to come from:
- Condition ordering in nested `if` (e.g., check `lo` first vs. `hi` first)
- Loop variable name choice (e.g., `i` vs. `idx`)
- Mutable variable naming in fibonacci (e.g., `a`/`b` vs. `prev`/`curr`)

Python variance comes from variable names, docstrings, comments, blank lines, and algorithmic style.

---

## Methodology Notes

- NAIL outputs are **normalized to JCS canonical form** (`json.dumps(sort_keys=True, separators=(',', ':'))`) before hashing — this removes key-order variance, which is not semantically meaningful.
- NAIL outputs are also **type-checked** using the NAIL interpreter's `Checker`. Invalid outputs are flagged but still counted in hash diversity.
- Python outputs are hashed as-is (raw text). No normalization is applied, which makes the comparison conservative (favoring Python).
